How RADIUS Is Put Together

RADIUS is the CRM the ACPD team uses to shepherd Circle Leaders — profiles, follow-ups, boards, scorecards, and weekly circle summaries. It's one Next.js codebase that serves three different audiences from three hostnames, stores everything in Supabase, and leans heavily on Church Community Builder (CCB) as the source of truth for people, groups, and attendance.

The big picture

Every request enters through one Next.js 14 app hosted on Netlify. Middleware looks at the hostname to decide which "face" of the app you get. All three faces share the same Supabase database and the same CCB integration.

The CCB connection, up close

This is the most intricate part of RADIUS. CCB is read-mostly from the app's perspective (the writes are narrow: attendance, notes, group membership). Every CCB call flows through a gateway that enforces a daily budget so RADIUS never blows through CCB's rate limits.

~50 API routes
/app/api/ccb/*, campaigns, toolkits, event-summary tracker, admin tools.
Cron jobs
sync-attendance (daily) and prewarm-circle-summary (daily) pull CCB data on a schedule.
→
ccb-api-gateway.ts
Atomic daily call budget — default 9,500 calls/day, reset at midnight Central via a Postgres RPC. Logs every request and raises alerts on 429s or high usage.
Telemetry tables
ccb_api_requests, _rate_limits, _daily_status, _alerts — visible in the in-app CCB usage dashboard.
→
v1 — XML API (legacy)
ccb-client.ts. Basic-auth XML services: group profiles & participants, event/attendance profiles, individual search, queues, forms, notes, calendar listings.
v2 — REST API (migration target)
ccb-v2-client.ts + OAuth manager. Authorization-code grant once, then a stored refresh token mints ~2-hour access tokens (ccb_oauth_tokens). Powers OAuth routes, scheduling, group rosters, campaign reconcile, and the whole Teams Toolkit data layer.

Reads are a hybrid: some flows hit CCB live per request, but anything hot gets cached into Supabase so pages stay fast and the budget stays safe — individual profiles (ccb_individual_profiles), group events (ccb_group_events_cache), circle rosters (circle_roster_cache), and custom-field dictionaries (ccb_custom_field_definitions / _options, harvested from v1 so v2's opaque field IDs can be resolved without extra calls). Sync bookkeeping lives in ccb_week_sync_log, ccb_orphan_summaries, and related audit tables.

Where the migration stands: v1 and v2 run side by side today. v2 already carries OAuth, scheduling categories, rosters, campaign reconciliation, and Teams Toolkit; the bulk of circle-summary and attendance sync still rides v1 XML. The v2 clients deliberately mirror v1 method shapes so flows can be moved over one at a time.

Hosting & deployment notes

Three Netlify sites run the same repo; the hostname decides which portal a visitor sees.

  • Main site — the full RADIUS app. Pushing to main auto-deploys; all cron jobs run here.
  • Circle Leader Toolkit site — serves the leader portal at its host root and exposes only /api/circle-leader-toolkit/*. Cron disabled.
  • Teams Toolkit site — same pattern for team leaders, behind a feature flag. Cron disabled.
  • Magic links are signed on the main site and verified on the toolkit sites, so LEADER_SESSION_SECRET must be byte-identical across all three.
  • The toolkit sites currently have Netlify builds stopped — shipping a toolkit fix means manually unstopping, building, and re-stopping those sites.
The one rule that protects everything: the Supabase service-role key and all CCB/AI/Resend credentials are server-side only. The browser only ever holds the public Supabase anon key — row-level security does the real enforcement.
